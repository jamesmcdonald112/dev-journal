**What tokens are**

- Design tokens are named values: spacing, font sizes, colors, radii, shadows.
    
- In web CSS they’re usually **CSS variables** in tokens.css.

### **The tokens you will almost always have**

- **Colors** (brand, text, background, muted)
    
- **Font family** (and maybe sizes/line-heights)
    
- **Spacing scale** (often keep Tailwind default, but you may tweak)
    
- **Border radius**
    
- **Shadows**
    
- **Breakpoints** (usually keep default)
  

## **Copy/paste vs real scaling**

  

Copy/paste per project is fine up to ~10–20 sites. At **100**, it’s a maintenance trap.

  

You want one of these models:

  

### **Model A (best): Shared “starter” + per-client tokens only**

- You maintain a single base repo/template
    
- Each client site only overrides a small token file:
    
    - tokens.css or brand.json
        
    
- Everything else is shared patterns/components
    

  

### **Model B: Monorepo**

- One codebase
    
- 100 site configs
    
- Shared components, shared updates
    
- Client-specific tokens per site
    

  

### **Model C: Copy/paste projects (avoid)**

- 100 repos
    
- Every improvement requires 100 updates
    
- You will stop updating them because it’s too much work
    

  

## **Practical recommendation for you**

  

Use Tailwind + tokens via CSS variables:

- Keep Tailwind for layout/spacing/structure
    
- Put branding in tokens.css like:
    
    - --color-brand, --color-bg, --radius, etc.
        
    
- In Tailwind config, map colors to CSS variables (so each client is just swapping variables)
    

  

This gives you:

- Tailwind speed
    
- Per-client theming with minimal edits
    
- Centralized updates
