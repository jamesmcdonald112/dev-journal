### **1) SVGs are text files**

  

An .svg isn’t “just an image” — it’s XML.

  

To view/edit it:

- Right-click the SVG file in Cursor / VS Code
    
- **Open With → Text Editor**
    
- You’ll see `<svg>...</svg>` markup
    

---

### **2) Make the SVG colorable**

  

If you want Tailwind/CSS to control the color, the SVG must use currentColor.

  

Change this:

```
<svg fill="#000000" ...>
```

To this:

```
<svg fill="currentColor" ...>
```

If it’s a stroke icon, change:

```
stroke="#000000"
```

to:

```
stroke="currentColor"
```

---

### **3) Why ?raw is required**

  

Normally, importing an SVG gives you a file URL.

  

?raw tells Astro/Vite:

  

> “Import the **contents of the file as a string**.”

  

Example:

```
import companyLogo from "../assets/logo/companyLogo.svg?raw";
```

Now companyLogo is a string containing <svg>...</svg>.

---

### **4) Why we use set:html={companyLogo}**

  

Astro does **not** render strings as HTML by default.

  

Without set:html, you’d literally see <svg>...</svg> as text.

```
<span set:html={companyLogo} />
```

This tells Astro:

  

> “Inject this string as real HTML markup.”

---

### **5) Why we use a `<span>` wrapper**

  

Because we’re **injecting** an `<svg>` string, we need a container.

  

The wrapper:

- controls size
    
- controls color
    
- controls layout
    

  

It can be a span or div.

We use span because it fits naturally inside text/navigation.

---

### **6) Why the SVG must be block (or inside flex)**

  

SVGs are **inline elements by default**.

  

Inline elements:

- align to the text **baseline**
    
- interact with line-height
    
- often create **mystery vertical space** in headers/navbars
    

  

This is what caused your header to grow unexpectedly.

  

To avoid this, we do **one of these** (both are valid):

  

#### **Option A — make the SVG block-level (recommended)**

```
[&_svg]:block
```

This removes baseline alignment issues.

  

#### **Option B — put it inside a flex container**

```
<a class="inline-flex items-center">
```

Flex layout removes inline baseline behavior entirely.

  

**In practice, we do both** for safety.

---

### **7) What this means**

```
[&_svg]:h-full
[&_svg]:w-full
[&_svg]:block
```

This is Tailwind’s **arbitrary selector** syntax.

  

It targets the <svg> _inside_ the wrapper:

- h-full → SVG fills wrapper height
    
- w-full → SVG fills wrapper width
    
- block → no inline/baseline weirdness
    

  

Why this is needed:

- Many SVGs ship with huge intrinsic sizes (e.g. 800×800)
    
- Without forcing sizing, they ignore the wrapper
    
- This guarantees predictable sizing
    

---

### **8) The correct, stable pattern**

```
<a href="/" class="inline-flex items-center p-1.5 -m-1.5">
  <span
    set:html={companyLogo}
    class="h-8 w-8 text-accent [&_svg]:h-full [&_svg]:w-full [&_svg]:block"
    aria-hidden="true"
  />
</a>
```

Key points:

- inline-flex items-center → clean vertical alignment
    
- wrapper controls size + color
    
- SVG inherits color via currentColor
    

---

### **9) How you change the color**

  

Because the SVG uses currentColor, just change text color:

```
text-white
text-accent
text-textOnDark
```

No SVG edits required after that.

---

### **Summary (mental model)**

- SVG color comes from **CSS**, not the file
    
- SVG sizing comes from **the wrapper**
    
- Inline SVGs cause layout bugs → **use block or flex**
    
- ?raw + set:html is the clean Astro way
    

  

This is how professionals do inline, themeable SVGs in component-based layouts.
